# test-engine

**Responsabilidad única:** Dado el output del ticket-analyst, descubrir sessions existentes,
ejecutarlas con Jest y devolver resultados estructurados. No genera tests nuevos. No escribe en Jira.

---

## Input

Recibe el Pipeline Context completo — siempre leyendo desde disco el JSON actualizado
por ticket-analyst en `pipeline-logs/completed/<TICKET_KEY>.json`.

Campos que consume:
```json
{
  "pipeline_id": "pipe-YYYYMMDD-NNN",
  "ticket_analyst_output": {
    "ticket_key": "NAA-XXXX",
    "classification": {
      "domain": "post",
      "module": "ai-post",
      "action_type": "regression_test",
      "testable": true,
      "confidence": "high"
    },
    "test_hints": ["..."],
    "acceptance_criteria": ["..."]
  }
}
```

> Si `classification.testable = false` → saltar a TE-8 con `sessions_found: false`.

---

## Paso TE-1: Verificar disponibilidad del grid

Antes de cualquier otra acción, verificar que Docker Selenium Grid responde:

```bash
curl -s http://localhost:4444/status | python3 -c "import sys,json; d=json.load(sys.stdin); print('ready' if d['value']['ready'] else 'not_ready')"
```

| Resultado | Acción |
|-----------|--------|
| `ready: true` | Continuar con TE-2 |
| `ready: false` o timeout | Registrar en `test_engine_output.result: "error"`, `error_type: "infra"`, detener y escribir Pipeline Context con `stage_status: "failed"` |

> **Distinción infra vs. app:** Un error de grid (`ECONNREFUSED`, timeout de Selenium) es
> siempre `error_type: "infra"`. Un assertion failure o excepción en el test es `error_type: "app"`.

---

## Paso TE-2: Calcular `execution_id`

Leer los archivos existentes en `pipeline-logs/` con patrón `results-<TICKET_KEY>-exec-*.json`
y calcular el siguiente número secuencial:

```
execution_id = "exec-YYYYMMDD-<N+1>"   (ej: exec-20260414-002 si ya existe exec-20260414-001)
```

Si no existen ejecuciones previas para ese ticket y fecha: `exec-YYYYMMDD-001`.

---

## Paso TE-3: Determinar modo de operación

| Condición | Modo |
|-----------|------|
| Input no especifica sessions a correr | `discover_and_run` (default) |
| Orchestrator ya conoce los session paths | `run_existing` |

En `run_existing`, saltar TE-4 y TE-5 — ir directo a TE-6 con los paths provistos.

---

## Paso TE-4: Discovery — buscar sessions en `test-map.json`

Leer `.claude/pipelines/test-engine/references/test-map.json`.

**Precedencia de matching** (orden estricto):

1. **Exact module match:** `classification.module` == key en `modules{}` (ej: `"ai-post"`)
2. **Domain match:** `classification.domain` == key en `modules{}` (ej: `"post"`)
3. **Keyword match:** algún `test_hint` o `acceptance_criteria` contiene keyword del módulo
4. **Component match:** `ticket_analyst_output.component_jira` matchea `component_jira` del módulo

Si ningún match → `sessions_found: false`. Saltar a TE-8.

Si hay match → registrar `matched_module` y `matched_sessions[]` con todos los paths del módulo.

---

## Paso TE-5: Verificar que los archivos existen en disco

Para cada path en `matched_sessions[]`:

```bash
# Verificar existencia física
ls <session_path>
```

Si algún path en `test-map.json` no existe en disco → remover de la lista y loguearlo.
Si la lista queda vacía tras las verificaciones → `sessions_found: false`. Saltar a TE-8.

---

## Paso TE-6: Construir y ejecutar comando Jest

### Master / Testing (default)
```bash
NODE_OPTIONS='--experimental-vm-modules' USE_GRID=true IS_HEADLESS=true \
  node node_modules/.bin/jest {TestName} \
  --json --outputFile=pipeline-logs/results-{ticket}-{execution_id}.json
```

### Dev_SAAS (pasar TESTING_URL inline — nunca modificar .env)
```bash
NODE_OPTIONS='--experimental-vm-modules' USE_GRID=true IS_HEADLESS=true \
  TESTING_URL="{dev_saas_url}" \
  node node_modules/.bin/jest {TestName} \
  --json --outputFile=pipeline-logs/results-{ticket}-{execution_id}.json
```

> **WSL2:** `npm` y `npx` resuelven al binario de Windows (`/mnt/c/Program Files/nodejs/`).
> Usar siempre `node node_modules/.bin/jest` directamente. `cross-env` es innecesario en bash.

**Resolución de `{TestName}`:** Nombre del archivo de session sin path ni extensión.

| Session file | TestName |
|---|---|
| `sessions/post/NewAIPost.test.ts` | `NewAIPost` |
| `sessions/video/NewYoutubeVideo.test.ts` | `NewYoutubeVideo` |
| `sessions/tags/AutoGenerated_NAA-XXXX.test.ts` | `AutoGenerated_NAA-XXXX` |

Si hay múltiples sessions en `matched_sessions[]`: ejecutar una por vez (no en paralelo).
`MAX_INSTANCES=1` está implícito — no agregar como flag.

---

## Paso TE-7: Parsear resultado del JSON output de Jest

Leer `pipeline-logs/results-{ticket}-{execution_id}.json` después de la ejecución.

Estructura que provee Jest con `--json`:
```json
{
  "numPassedTests": 1,
  "numFailedTests": 0,
  "testResults": [
    {
      "testFilePath": "sessions/post/NewAIPost.test.ts",
      "status": "passed",
      "testResults": [
        {
          "fullName": "NewAIPost",
          "status": "passed",
          "duration": 42000,
          "failureMessages": []
        }
      ]
    }
  ]
}
```

**Mapeo a `results[]` del output:**

Para cada `assertionResult` en `testResults[*].testResults`:

```json
{
  "session_file": "<testFilePath relativo desde raíz del repo>",
  "session_name": "<basename sin extensión del testFilePath>",
  "status": "pass | fail | error",
  "duration_ms": "<assertionResult.duration>",
  "failure_messages": "<assertionResult.failureMessages[]>",
  "mapped_to_hint": "<test_hint[i] del ticket-analyst si existe, si no: null>"
}
```

> `status: "error"` cuando Jest no pudo correr el archivo (error de sintaxis, import fallido).
> `status: "fail"` cuando el test corrió pero una assertion falló.

**Pre-computar `failure_summary`** (solo si `numFailedTests > 0`):
```json
{
  "test_name": "<fullName del primer test fallido>",
  "error_message": "<primera línea de failureMessages[0]>",
  "source": "<extraer de la primera línea del stack que contenga src/>",
  "stack": "<primeras 5 líneas del stack completo>"
}
```

## Paso TE-8: Escribir `test_engine_output` en el Pipeline Context

Leer el Pipeline Context desde `pipeline-logs/completed/<TICKET_KEY>.json`, agregar
`test_engine_output` con el schema unificado y reescribir el archivo completo:

```json
"test_engine_output": {
  "pipeline_id": "<pipeline_id del contexto>",
  "ticket_key": "NAA-XXXX",
  "execution_id": "exec-YYYYMMDD-NNN",
  "executed_at": "<ISO timestamp>",
  "mode_used": "discover_and_run | run_existing",
  "environment": "master | dev_saas | testing",
  "grid": true,
  "headless": true,
  "sessions_found": true,
  "result": "passed | failed | error",
  "total_tests": 1,
  "passed": 1,
  "failed": 0,
  "results": [
    {
      "session_file": "sessions/post/NewAIPost.test.ts",
      "session_name": "NewAIPost",
      "status": "pass | fail | error",
      "duration_ms": 42000,
      "failure_messages": [],
      "mapped_to_hint": "..."
    }
  ],
  "failure_summary": null,
  "console_errors_detected": [],
  "jest_output_path": "pipeline-logs/results-NAA-XXXX-exec-NNN.json"
}
```

Actualizar también:
- `stage: "test-engine"`
- `stage_status: "completed" | "failed"`
- Agregar entrada en `step_log[]`:
  ```json
  { "stage": "test-engine", "timestamp": "<ISO>", "status": "completed | failed", "duration_ms": <N> }
  ```

> Cuando `sessions_found: false`: escribir `test_engine_output` con `sessions_found: false`,
> `result: "error"`, `error_type: "no_sessions"`. El orchestrator invoca `test-generator`
> en este caso — test-engine no hace nada más.

---

## Output del pipeline

El Pipeline Context final en `pipeline-logs/completed/<TICKET_KEY>.json` con:
- `stage: "test-engine"`, `stage_status: "completed" | "failed"`
- `test_engine_output` completo con schema unificado
- `pipeline-logs/results-<TICKET_KEY>-<execution_id>.json` creado en disco

---

## Manejo de errores

| Error | `result` | `error_type` | Acción |
|-------|----------|--------------|--------|
| Grid no disponible | `error` | `infra` | Escribir output, marcar `stage_status: failed`, detener |
| Session no existe en disco | — | — | Remover de lista; si lista vacía → `sessions_found: false` |
| Jest falla por error de sintaxis/import | `error` | `app` | Capturar stderr, escribir en `failure_summary` |
| Jest retorna `numFailedTests > 0` | `failed` | `app` | Normal — parsear y escribir failure_summary |
| `outputFile` no creado tras ejecución | `error` | `infra` | Jest no corrió correctamente — reportar con stderr |
| Pipeline Context no encontrado | — | — | Fallar con error explícito — ticket-analyst no completó |

## Consideraciones GitHub Actions

- **Grid:** En CI, reemplazar `http://localhost:4444` con la URL del Selenium Grid del workflow
  (`GRID_URL` secret). La verificación de TE-1 aplica igualmente.
- **Env vars:** `USE_GRID`, `IS_HEADLESS`, `ADMIN_USER`, `ADMIN_PASS`, `EDITOR_USER`,
  `EDITOR_PASS`, `BASIC_AUTH_USER`, `BASIC_AUTH_PASS` se toman del `.env` del repo
  o como secrets de GitHub Actions — nunca hardcodear.
- **TESTING_URL para Dev_SAAS:** Pasar como variable inline en el comando (§11.5 architecture
  doc) — nunca modificar el `.env`.
- **Artifacts:** Subir `pipeline-logs/results-*.json` como artifact del job para trazabilidad.
  El `jest_output_path` en el output apunta al archivo exacto.
- **Timeout:** Configurar `timeout-minutes: 10` en el step de test-engine para sessions largas.
- **Re-run safety:** `execution_id` es secuencial y único por fecha — un re-run del job genera
  un `exec-YYYYMMDD-002` sin pisar el anterior.
