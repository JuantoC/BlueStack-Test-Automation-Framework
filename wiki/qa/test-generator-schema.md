# test-generator Schema

> Fuente canónica del contrato de entrada/salida del agente test-generator. Referenciada por `test-generator.md` y `qa-orchestrator.md`.

## Input schema (desde qa-orchestrator, ORC-4)

```json
{
  "ticket_key": "NAA-XXXX",
  "domain": "post | video | images | auth | ai-post",
  "acceptance_criteria": [...],
  "test_hints": "descripción breve del flujo a testear",
  "pom_paths": ["src/pages/..."]
}
```

| Campo | Descripción |
|-------|-------------|
| `ticket_key` | Key del ticket Jira (ej. NAA-4465) |
| `domain` | Dominio del módulo clasificado por ticket-analyst. Valores válidos: `"post"`, `"video"`, `"images"`, `"auth"`, `"ai-post"` |
| `acceptance_criteria` | Array completo de criterios del ticket-analyst, con `criterion_scope` |
| `test_hints` | Descripción breve del flujo, tomada de `classification.test_hints` |
| `pom_paths` | Paths de Page Objects relevantes. Derivado automáticamente por ORC-4.1 desde los paths del módulo en `test-map.json` (campo `pom_paths[]` de la entry del módulo clasificado). Si el módulo no tiene paths mapeados, ORC pasa array vacío. |

## Output schema (retornado al qa-orchestrator)

```json
{
  "status": "auto_generated | failed",
  "test_path": "sessions/{domain}/AutoGenerated_{ticket_key}.test.ts",
  "dry_run_result": "pass | fail | infra_error",
  "notes": "..."
}
```

| Campo | Descripción |
|-------|-------------|
| `status` | `"auto_generated"` = test generado y dry-run completado; `"failed"` = fallo en algún paso |
| `test_path` | Path relativo del archivo `.test.ts` generado |
| `dry_run_result` | `"pass"` = exit 0 sin `error TS`; `"fail"` = error de compilación TypeScript; `"infra_error"` = Docker Grid no disponible |
| `notes` | Detalle libre sobre el resultado o errores encontrados |

## Bloque `test_generator_output` en el Execution Context

Escrito en `pipeline-logs/active/<ticket_key>.json` al completar TG-6.

```json
"test_generator_output": {
  "status": "auto_generated | failed",
  "test_path": "sessions/{domain}/AutoGenerated_{ticket_key}.test.ts",
  "dry_run_result": "pass | fail | infra_error",
  "test_map_updated": true,
  "notes": "..."
}
```

## Entrada provisional en test-map.json (TG-5)

Formato de la entrada que test-generator agrega al archivo `.claude/pipelines/test-engine/references/test-map.json`:

```json
{
  "module": "{domain}",
  "sessions": [
    {
      "file": "sessions/{domain}/AutoGenerated_{ticket_key}.test.ts",
      "description": "Auto-generado para {ticket_key} — requiere revisión manual",
      "validated": false,
      "auto_generated": true,
      "ticket": "{ticket_key}",
      "generated_at": "<ISO-8601>"
    }
  ]
}
```

> `validated: false` es crítico. test-engine no incluye estos tests en ejecuciones automáticas hasta que el campo sea `true`. Se habilita cuando el test es commiteado con `[validated]` en el mensaje de commit.

## Cabecera obligatoria del archivo generado

```typescript
// @auto-generated: true
// @ticket: {ticket_key}
// @validated: false  // cambiar a true después de revisión manual
// @target-env: master  // si Ambiente destino fue especificado
// @default-role: editor
```

## Reglas de generación de steps por tipo de criterio

| `automatable` | `criterion_scope` | Qué genera |
|---------------|-------------------|------------|
| `true` | `"ui"` | Steps y aserciones Selenium normales |
| `false` | cualquiera | Comentario `// MANUAL:` en lugar de assertion |
| `true` o `false` | `"vfs"` o `"backend_data"` | Comentario `// MANUAL: requiere verificación backend` |
